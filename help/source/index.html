Overview

The Advanced Search Panel is a powerful QGIS plugin that adds configurable search capabilities to your QGIS interface. It supports multiple API providers, custom search configurations, and provides a modern dockable search interface.

Key Features

  • Multi-Provider Search: Search through multiple API endpoints simultaneously
  • Layer-Based Search: Search within your project layers and feature attributes
  • Pre-configured Templates: Quick setup for popular geocoding services
  • Flexible Configuration: Support for GET/POST requests, custom headers, and authentication
  • Import/Export: Backup and share configurations
  • Regular Expression Filtering: Fine-tune search results

Installation

To install the Advanced Search Panel plugin:

  1. Download the plugin from the QGIS Plugin Repository or from the source repository
  2. Extract the plugin to your QGIS plugins directory:
    • Windows: %APPDATA%\QGIS\QGIS3\profiles\default\python\plugins\
    • Linux: ~/.local/share/QGIS/QGIS3/profiles/default/python/plugins/
    • macOS: ~/Library/Application Support/QGIS/QGIS3/profiles/default/python/plugins/
  3. Restart QGIS
  4. Enable the plugin in Plugins → Manage and Install Plugins
  5. The search panel will appear in your QGIS interface

Getting Started

After installation, you'll see a new search panel in your QGIS interface. The panel can be docked anywhere in the QGIS window for optimal workflow integration.

Basic Usage

  1. Open Configuration: Click the gear/settings icon in the search panel
  2. Add Provider: Use a pre-configured template or create a custom provider
  3. Search: Type in the search box and press Enter
  4. View Results: Click on any result to zoom to its location

Configuration

The plugin configuration is accessed through the settings button in the search panel. The configuration dialog has two main tabs:

Search Providers Tab

Configure your search providers including API endpoints, authentication, and result parsing.

  • Add Provider: Create new search providers
  • Edit Provider: Modify existing providers (double-click or use Edit button)
  • Remove Provider: Delete unwanted providers
  • Reorder: Change search priority with Move Up/Down buttons
  • Import/Export: Backup and share configurations

General Settings Tab

Configure global plugin behavior:

  • Search Behavior: Stop on first result, result limits, timeouts
  • Zoom Settings: Configure zoom behavior for different coordinate systems

API Provider Templates

The plugin includes pre-configured templates for popular geocoding services:

Available Templates

  • Google Places API: High-quality place search with detailed information
  • Nominatim (OpenStreetMap): Free geocoding service (no API key required)
  • Mapbox Geocoding: Fast, customizable search service

Using Templates

  1. Click "Add Provider" in the Search Providers tab
  2. Select a template from the "Type" dropdown
  3. Follow the setup instructions dialog
  4. Add your API credentials where indicated
  5. Save and test the configuration

Layer-Based Providers

Layer-based providers allow you to search within the features of layers loaded in your QGIS project. This is useful for searching local datasets, databases, or any other layer data without requiring an external API.

Setting Up a Layer Provider

  1. Click "Add Provider" in the Search Providers tab
  2. Select "Layer" from the "Type" dropdown
  3. Give your provider a descriptive name (e.g., "Address Points", "Property Database")
  4. Select the layer you want to search from the "Layer" dropdown
  5. Configure the search fields and match mode (see below)
  6. Optionally add a Regex Filter to control when this provider activates
  7. Save the configuration

Search Fields

The Search Fields input specifies which attribute fields in your layer should be searched. You can specify multiple fields separated by commas.

Examples:

  • name - Search only the "name" field
  • address,street_name,postcode - Search across multiple fields
  • reference_number,uprn - Search identifiers

When multiple fields are specified, the search will return results if the search term is found in any of the fields (OR logic).

Search Mode: Wildcard vs Exact Match

The search mode determines how your search term is matched against the field values:

Wildcard Match (Default)

Wildcard Match performs a case-insensitive "contains" search. The search term can appear anywhere within the field value - at the beginning, middle, or end.

  • Search term: oak
  • Will match:
    • "Oak Street" (beginning)
    • "123 Oakland Drive" (middle)
    • "Royal Oak" (end)
    • "The Grand Oak Hotel" (middle)
  • Won't match: "Maple Avenue"

Technical detail: Uses SQL ILIKE '%search_term%' - the % wildcards match any characters before and after the search term.

Use wildcard match when:

  • Users might search for partial addresses or names
  • You want flexible, forgiving search behavior
  • Searching text fields with variable content
  • Users don't know the exact spelling or full value

Exact Match

Exact Match performs a case-insensitive comparison where the search term must match the entire field value exactly (ignoring case).

  • Search term: high street
  • Will match:
    • "High Street"
    • "HIGH STREET"
    • "high street"
  • Won't match:
    • "High Street North" (extra text)
    • "123 High Street" (extra text)
    • "Highland Road" (different text)

Use exact match when:

  • Searching for unique identifiers (IDs, reference numbers, UPRNs)
  • Field values are standardized and complete
  • You want to prevent partial matches from cluttering results
  • Searching categorical fields with discrete values
  • Searching date fields where you want an exact date match

Using Regex Filters with Layer Providers

The Regex Filter field controls when a provider is activated, not how results are matched. If specified, the search term must match the regex pattern for the provider to run at all.

This is useful for creating specialized providers that only activate for specific types of input.

Regex Filter Examples

UK Postcode Search Only
  • Regex Filter: ^[A-Z]{1,2}\d{1,2}\s?\d[A-Z]{2}$
  • Activates for: "BH1 1AA", "SW1A 1AA", "M1 1AE"
  • Ignores: "High Street", "123", regular text searches

Use this to create a dedicated postcode search provider that only runs when the user enters a valid UK postcode format.

Numeric Reference Number Only
  • Regex Filter: ^\d{6,10}$
  • Activates for: "123456", "1234567890"
  • Ignores: "Oak Street", "abc123", any text

Use this for searching property reference numbers or UPRNs without triggering on address searches.

Date Format Search (YYYY-MM-DD)
  • Regex Filter: ^\d{4}-\d{2}-\d{2}$
  • Activates for: "2024-01-15", "2023-12-31"
  • Ignores: "15/01/2024", "January 15", any other format

Use this with exact match mode to find features with a specific date value.

Layer Provider Examples

Example 1: Address Search with Wildcard

  • Name: "Address Lookup"
  • Type: Layer
  • Layer: "Property_Addresses"
  • Search Fields: full_address,street_name
  • Search Mode: Wildcard
  • Regex Filter: (leave empty - allows any search)

This configuration allows users to search for partial addresses. Searching "oak" would find "Oak Avenue", "123 Oakland Drive", "Royal Oak Court", etc.

Example 2: Property Reference Search with Exact Match

  • Name: "Property ID Search"
  • Type: Layer
  • Layer: "Property_Database"
  • Search Fields: uprn,reference_number
  • Search Mode: Exact
  • Regex Filter: ^\d{6,12}$

This configuration only activates when the user enters a 6-12 digit number, and requires an exact match. Perfect for unique property identifiers.

Example 3: Postcode-Only Search

  • Name: "UK Postcode Search"
  • Type: Layer
  • Layer: "Addresses"
  • Search Fields: postcode
  • Search Mode: Exact
  • Regex Filter: ^[A-Z]{1,2}\d{1,2}\s?\d[A-Z]{2}$

This provider only activates for valid UK postcode formats and requires an exact match. Prevents the postcode provider from running on general address searches.

Example 4: Date Search

  • Name: "Planning Application by Date"
  • Type: Layer
  • Layer: "Planning_Applications"
  • Search Fields: submission_date,decision_date
  • Search Mode: Exact
  • Regex Filter: ^\d{4}-\d{2}-\d{2}$

This searches for planning applications submitted or decided on a specific date. Users must enter dates in YYYY-MM-DD format (e.g., "2024-01-15").

Note: The current implementation only supports searching for exact date matches, not date ranges (e.g., "after 2024-01-01"). For date range queries, you would need to use QGIS's built-in attribute table filtering or create a custom query.

Performance Considerations

When working with large layers, consider these tips:

  • Limit search fields: Only include fields that users need to search
  • Use exact match for IDs: Reduces the number of features that need to be checked
  • Add regex filters: Prevent searches from running on inappropriate terms
  • Enable spatial indexing: Ensure your layer has a spatial index for better performance
  • Use the "Stop on Result" option: Prevent unnecessary searches once a match is found
  • Consider multiple specialized providers: Instead of one general provider, create specific providers for postcodes, references, addresses, etc., each with appropriate regex filters

Import/Export Configuration

The plugin supports exporting and importing provider configurations for backup and sharing.

Exporting

  1. Open the configuration dialog
  2. Go to the "Search Providers" tab
  3. Click the "Export..." button
  4. Choose a location to save the JSON file

Importing

  1. Open the configuration dialog
  2. Go to the "Search Providers" tab
  3. Click the "Import..." button
  4. Choose between "Replace All" or "Merge" mode
  5. Select the JSON file to import

Note: Layer-based providers in exported configurations reference layers by ID. When importing on another system, ensure the referenced layers exist in the target project or the layer provider will need to be reconfigured.

Advanced Features

Custom API Providers

Create custom providers for any REST API that returns location data:

  • URL Templates: Use {search_term} placeholder
  • HTTP Methods: Support for GET and POST requests
  • Custom Headers: Authentication and content type headers
  • POST Body Templates: JSON body templates with search term substitution
  • Result Parsing: Configurable JSON path extraction

Parsing results

Parsing results from JSON requires JSONPath expressions to extract relevant fields. Use a service like https://jsonpath.com/ to help you.

There are five fields you can fill in under Result Parsing:

Results Array Path (optional)

This is the path to the array of results. Some services include other metadata about the result set at the top level, and the actual results you want seperately. This parameter lets you specify where those results can be found.

Name Path

This is the JSONPath to a field which will be used as the display name for the result, generally a full address field

Latitude Path

This is the JSONPath to the field containing the latitude coordinate of the result.

Longitude Path

This is the JSONPath to the field containing the longitude coordinate of the result.

Bounding Box Field/Template

This is the JSONPath to the field containing the bounding box coordinates of the result.

You can either specify a single field that contains the bounding box as an array of coordinates, or you can specify separate fields for the north, south, east, and west coordinates.

  • Single field: e.g. 'bbox' (if API returns [west,south,east,north])
  • Template: '{west},{south},{east},{north}' or '{minX},{minY},{maxX},{maxY}'

Note: You must have either the bounding box field or both the latitude and longitude fields filled in for the search to work properly.

Authentication

Multiple authentication methods supported:

  • API Keys: In headers or URL parameters
  • QGIS Authentication: Use QGIS authentication configuration system
  • Custom Headers: Any authentication method supported by the API

Regular Expression Filtering

Make a search only happen when the search term matches specific patterns, such as:

  • UK Postal codes
  • Reference numbers in your data
  • Coordinate pairs
  • Date formats
  • Anything else you can think of!

Important: The regex filter validates the search term (what the user types), not the feature data in your layer. Use it to control which provider activates based on user input.

Troubleshooting

Common Issues

No Search Results

  • Check API key configuration
  • Verify internet connection
  • Check provider is enabled
  • For layer providers: verify the layer is loaded and field names are correct
  • For regex filters: ensure your search term matches the pattern
  • Review QGIS logs for errors

Authentication Errors

  • Verify API key is correct and active
  • Check quota limits on API account
  • Ensure required services are enabled
  • Verify header format and values

Parsing Errors

  • Check Results Path configuration
  • Verify field mapping names match API response
  • Test with simple search terms first
  • Compare API response format with configuration

Layer Provider Issues

  • Verify the layer is loaded in the current QGIS project
  • Check that field names in "Search Fields" exactly match layer attribute names (case-sensitive)
  • Ensure the layer has features and isn't empty
  • Try both wildcard and exact match modes to diagnose the issue
  • Check for special characters in field values that might affect matching
  • If using a regex filter, test your pattern at regex101.com
  • Remember: wildcard matches anywhere in the text (beginning, middle, or end)

Regex Filter Not Working

  • Test your regex pattern using online tools like regex101.com
  • Remember to escape special characters (e.g., use \. for a literal period)
  • Use ^ and $ anchors to match the entire search term
  • Regex patterns are case-sensitive unless you use flags like (?i)
